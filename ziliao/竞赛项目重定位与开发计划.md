# 全国大学生物联网设计竞赛项目重定位与开发计划

## 1. 背景与问题判断

当前项目最初参考了“光伏发电板自动清洁控制系统”毕业设计材料，并以“基于物联网与图像识别的光伏发电板智能清洁控制系统”作为竞赛报名方向。经过对现有 ESP32-IDF 工程、论文资料和报名材料的核对，发现原报名材料中的系统边界明显大于当前代码工程的实际能力。

原报名材料主要承诺完整的“感知-决策-执行-云端管理”闭环系统，包括电机、水泵、4G、MQTT、Spring Boot、MySQL、Redis、Web 管理平台、远程控制和历史数据管理等内容。但当前工程的真实技术亮点集中在 ESP32-S3 本地视觉推理、摄像头采集、HTTP 视频流、标注图输出、OLED 显示和传感器接口。

因此，后续作品必须重新定位，避免继续沿用与实际工程不一致的“完整自动清洁系统”叙事。新的竞赛材料、设计文档和视频演示应围绕当前项目真实可落地的核心能力展开，并通过最小闭环补足执行联动和物联网通信演示。

## 2. 新作品定位

推荐作品名称：光伏板边缘AI监测系统

备选名称：

- 光伏板智能清洁决策终端
- 光伏板AI智检清洁系统
- 光伏板边缘智检终端

推荐定位：

> 本作品面向分布式光伏组件运维场景，设计并实现一套基于 ESP32-S3 的边缘 AI 智能监测终端。系统在设备端完成光伏板图像采集、YOLO11n 本地推理、环境与功率参数采集、脏污风险评估和清洁策略决策，并通过 WiFi/MQTT 与 Web 可视化界面实现状态展示、告警提示和执行器联动验证。

这个定位强调“边缘 AI + 多源感知 + 清洁决策 + 联动验证”，不再过度承诺完整机械清洁机器人、4G 云平台和多端应用生态。

## 3. 当前工程真实基础

当前 ESP32-IDF 工程已经具备以下基础能力：

- ESP32-S3 作为主控平台。
- OV5640 摄像头采集 320x320 JPEG 图像。
- YOLO11n INT8 模型通过 ESP-DL 在 ESP32-S3 本地运行。
- HTTP `/stream` 输出实时 MJPEG 视频流。
- HTTP `:8080/annotated` 输出检测标注 JPEG。
- 内置 `dashboard.html` Web 仪表盘。
- SHT20、TSL2584、MAX471 驱动文件已存在。
- SSD1306 OLED 驱动已存在。
- LED 状态指示模块已存在。
- ESP-IDF 构建、模型嵌入和分区配置已跑通。

当前需要修正的问题：

- OLED 当前显示随机模拟数据，真实 SHT20、TSL2584、MAX471 任务被注释。
- Web 端部分数据和远程控制按钮是模拟逻辑。
- 没有真实 MQTT 客户端和主题协议。
- 没有真实电机、水泵、继电器控制模块。
- 没有清洁评分、功率衰减、脏污面积占比融合决策逻辑。
- 原论文和报名材料中的 OV2640、4G、Spring Boot、MySQL、Redis 等内容与当前工程不一致。

## 5. 三层物联网架构修正版

### 5.1 感知层

感知层由 ESP32-S3 主控、OV5640 摄像头、SHT20 温湿度传感器、TSL2584 光照传感器、MAX471 电压/电流采集模块、继电器、电机和水泵组成。

主要功能：

- 采集光伏板表面图像。
- 本地运行 YOLO11n 模型识别脏污或缺陷。
- 采集温湿度、光照、电压/电流等环境和运行参数。
- 计算视觉脏污指数、功率衰减率和综合清洁评分。
- 根据策略控制电机和水泵。

注意事项：

- 摄像头型号应统一写 OV5640，除非实际硬件换成 OV2640。
- 传感器数据必须尽量使用真实读取值。
- 如果功率采集不稳定，可以在文档中说明当前原型采用 MAX471 采样和基准功率标定方式进行验证。

### 5.2 传输层

传输层推荐采用 WiFi + MQTT + HTTP 的组合。

- WiFi：利用 ESP32-S3 自带无线能力接入局域网或手机热点。
- MQTT：用于上传结构化状态数据和接收远程控制命令。
- HTTP：用于本地视频流、标注图和 Web 页面访问。

推荐 MQTT 主题设计：

```text
pv/device/{device_id}/telemetry     # 温湿度、光照、电压、电流、功率等遥测数据
pv/device/{device_id}/vision        # 检测类别、置信度、检测框、视觉脏污指数
pv/device/{device_id}/decision      # 综合评分、清洁建议、禁止清洁原因
pv/device/{device_id}/state         # 设备状态、执行器状态、联网状态
pv/device/{device_id}/cmd           # 远程控制命令
pv/device/{device_id}/config        # 阈值和策略配置
```

推荐 MQTT 消息示例：

```json
{
  "device_id": "pv-ai-001",
  "temp_c": 28.6,
  "humidity": 61.2,
  "lux": 23800,
  "voltage_mv": 5120,
  "current_ma": 180,
  "power_mw": 922,
  "dirt_index": 42.5,
  "power_loss": 18.0,
  "clean_score": 32.7,
  "decision": "brush_only",
  "state": "READY"
}
```

远程控制命令示例：

```json
{
  "cmd": "clean_start",
  "mode": "brush_only",
  "duration_s": 10
}
```

### 5.3 应用层

应用层优先只做 Web 端。

Web 端应展示：

- 实时摄像头画面。
- AI 标注检测画面。
- 温湿度、光照、电压/电流、功率等参数。
- 视觉脏污指数、功率衰减率、综合清洁评分。
- 清洁建议：无需清洁、滚刷清洁、喷淋清洁、延后清洁。
- 执行器状态：电机、水泵、继电器。
- 远程控制按钮：开始清洁、停止清洁、保存阈值。

不建议在本阶段同时开发微信小程序、安卓 APP 和复杂云后台。若时间充足，可补充一个 MQTT 数据看板；若时间不足，只需保证本地 Web 端稳定、清晰、能支撑视频演示。

## 6. 清洁决策算法设计

### 6.1 视觉脏污指数

根据 YOLO 检测结果计算检测框面积占比，并结合类别权重得到视觉脏污指数。

```text
A_i = 第 i 个检测框面积
A_total = 图像总面积
w_i = 类别权重
D_img = sum(A_i * w_i) / A_total * 100
```

推荐类别权重：

| 类别 | 权重 | 说明 |
| --- | ---: | --- |
| clean | 0.0 | 清洁状态不增加脏污指数 |
| dust | 1.0 | 灰尘按普通污染处理 |
| bird_drop | 1.5 | 鸟粪更影响发电，权重更高 |
| snow_covered | 1.3 | 积雪遮挡明显 |
| physical_damage | 2.0 | 物理损伤应告警，不一定清洁 |
| electrical_damage | 2.0 | 电损伤应告警，不应直接清洁 |

### 6.2 功率衰减率

可以先用标定基准功率和当前功率进行计算。

```text
P_now = 当前电压 * 当前电流
P_ref = 清洁状态或标定状态下的参考功率
P_loss = max(0, (P_ref - P_now) / P_ref * 100)
```

如果无法稳定测量实际电流，可在视频演示中使用“遮挡小太阳能板导致输出下降”的方式体现功率变化；文档中说明该算法支持由 MAX471 采样得到实时电压/电流并计算功率。

### 6.3 综合清洁评分

推荐使用简单线性融合，便于解释和实现。

```text
S = 0.6 * D_img + 0.4 * P_loss
```

推荐决策规则：

| 条件 | 决策 | 执行动作 |
| --- | --- | --- |
| S < 30 | 无需清洁 | 不动作，仅记录 |
| 30 <= S < 60 | 轻度污染 | 启动滚刷电机 |
| S >= 60 | 重度污染 | 启动水泵 + 滚刷电机 |
| 温度过低 | 禁止喷淋 | 只告警或延后清洁 |
| 光照过强 | 延后湿洗 | 可只滚刷或等待 |
| physical/electrical damage | 故障告警 | 不执行清洁，提示人工检修 |

### 6.4 视频中如何体现融合决策

建议录制四组演示：

1. 干净状态
   - 摄像头看到干净光伏板或干净图片。
   - 功率正常。
   - Web 显示“无需清洁”。

2. 只有视觉脏污
   - 摄像头看到灰尘或鸟粪测试图。
   - 功率衰减不明显。
   - Web 显示“视觉异常，持续观察或轻度清洁”。

3. 只有功率下降
   - 遮挡小太阳能板或改变光照。
   - 视觉画面未检测到明显脏污。
   - Web 显示“疑似光照变化或遮挡，暂不清洁”。

4. 视觉脏污 + 功率下降
   - 摄像头检测出脏污，同时功率下降。
   - 综合评分超过阈值。
   - 系统自动触发继电器、电机或水泵。

这四组演示能清楚体现系统不是单一指标触发，而是多源融合决策。

## 7. 执行器联动方案

### 7.1 最小硬件闭环

建议使用以下最小硬件：

- 1 个普通直流电机，模拟滚刷。
- 1 个水泵模块，模拟喷淋。
- 2 个继电器模块，分别控制电机和水泵。
- 独立电源给电机和水泵供电。
- ESP32-S3 GPIO 控制继电器输入端。

如果继电器为 5V 触发，应确认 3.3V GPIO 是否可稳定触发。若不稳定，需要三极管、MOS 管或光耦驱动模块。

### 7.2 推荐状态机

```text
IDLE           空闲
MONITORING     监测中
NEED_CLEAN     达到清洁条件
BRUSHING       滚刷清洁
SPRAYING       喷淋清洁
DONE           清洁完成
FAULT          故障或禁止清洁
```

状态转移示例：

```text
MONITORING -> NEED_CLEAN: clean_score 超过阈值
NEED_CLEAN -> BRUSHING: 轻度污染
NEED_CLEAN -> SPRAYING: 重度污染且环境允许喷淋
BRUSHING -> DONE: 达到设定时长
SPRAYING -> DONE: 达到设定时长
任意状态 -> FAULT: 温度过低、电损伤、人工停止或执行器异常
```

### 7.3 文档表述边界

建议使用“执行器联动验证”而不是“完整清洁机器人”。

推荐表述：

> 当前原型通过继电器、电机和水泵模块完成清洁执行链路验证，能够根据综合清洁评分自动触发滚刷或喷淋动作。实际工程部署时，可将该控制接口连接至光伏板清洁机构，实现更大功率执行器和机械结构的扩展。

## 9. 竞赛设计文档写作框架

竞赛模板要求正文包括：设计需求分析、特色与创新、功能设计、系统实现、其他内容、参考文献。建议按以下结构写。

### 9.1 设计需求分析

内容要点：

- 光伏板污染会造成发电效率下降。
- 人工巡检成本高，清洁时机难判断。

- 需要低成本、低功耗、可联网、可本地识别的边缘监测终端。

### 9.2 特色与创新

内容要点：

- 基于 ESP32-S3 的本地 YOLO11n 推理。
- 视频流与检测标注双通道展示。
- 视觉脏污指数与功率衰减率融合决策。
- 温湿度、光照约束下的清洁策略。
- 继电器、电机、水泵执行器联动验证。
- WiFi/MQTT 轻量物联网接入。

### 9.3 功能设计

建议分为：

- 图像采集与视觉识别功能。
- 环境参数采集功能。
- 功率参数采集功能。
- 清洁评分与决策功能。
- 执行器控制功能。
- Web 可视化与远程控制功能。
- MQTT 数据上传与命令下发功能。

### 9.4 系统实现

建议按三层架构写：

- 感知层实现。
- 传输层实现。
- 应用层实现。

同时加入关键流程：

```text
摄像头采集 -> JPEG 解码 -> YOLO 推理 -> 检测后处理 -> 视觉脏污指数
传感器采集 -> 功率计算 -> 功率衰减率
视觉脏污指数 + 功率衰减率 + 环境约束 -> 清洁决策
清洁决策 -> 继电器/电机/水泵 -> Web/MQTT 状态反馈
```

## 11. 风险与边界控制

### 11.1 不再承诺的内容

以下内容如果没有真实实现，不应写成“已完成”：

- EC200S-CN 4G 通信。
- PPP 拨号上网。
- MQTT 断线补发机制。
- Spring Boot 后端。
- MySQL/Redis 数据库。
- 微信小程序和安卓 APP。
- 完整机械清洁机器人。
- 实际大功率光伏清洁机构。
- 95% 以上识别准确率。

### 11.2 可以写成扩展方案的内容

- 户外部署可使用 4G、LoRa 或 ZigBee 网关。
- 应用层可扩展微信小程序和安卓端。
- 云端可扩展数据库和历史趋势分析。
- 执行器接口可连接更大功率电机和水泵。
- K230 可作为更高算力边缘节点扩展。

### 11.3 需要统一的技术口径

- 摄像头统一写 OV5640，除非实际换成 OV2640。
- 当前通信主线写 WiFi/MQTT，不写 4G 已完成。
- 当前应用主线写 Web，不写三端都已完成。
- 当前执行机构写联动验证，不写完整清洁机器人。
- 模型指标以实际训练和测试结果为准，不夸大。

## 13. 一句话总结

本项目后续不应再包装为“完整光伏自动清洁云平台”，而应重定位为“基于 ESP32-S3 的光伏板边缘 AI 监测与清洁决策终端”。在 7 月 27 日前，最关键的目标是跑通“真实感知数据 + 本地视觉推理 + 多源清洁评分 + 执行器联动 + Web 展示”的最小闭环。这个闭环真实、可拍、可解释，也最能体现物联网设计竞赛关注的系统集成能力。
