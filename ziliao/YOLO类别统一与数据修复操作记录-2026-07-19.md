## 2. 操作前的问题

### 2.1 三套类别定义不一致

操作前，工程中同时存在三套类别名称。

`yolov11/data.yaml` 和 `yolov11/train_and_export.py` 使用：

```text
0 bird
1 crack
2 dust
3 shadow
4 hotspot
5 panel
```

ESP32-S3 板端 `main/yolo_detect/yolo_detect.cpp` 使用：

```text
0 Bird
1 Cleaan
2 Dust
3 Electrical
4 Physical
5 Solar
```

竞赛重定位计划和项目 README 使用：

```text
0 bird_drop
1 clean
2 dust
3 electrical_damage
4 physical_damage
5 snow_covered
```

模型本身只输出数字 ID，并不知道业务名称。如果训练标签中的 ID 和板端解释不一致，同一个输出可能在训练报告和 ESP32 页面中显示成不同类别，后续清洁决策也会发生错误。

因此没有对约 2.98 万行标签执行全局 ID 替换，而是保留原始数字 ID，只统一名称和业务策略。

### 2.3 原训练/验证划分存在来源泄漏

Roboflow 导出的文件名通常包含：

```text
原图名称.rf.变体哈希.jpg
```

操作前检查结果：

```text
训练集原图来源：764 个
验证集原图来源：324 个
训练/验证共同来源：276 个
完全相同的文件哈希：0 个
```

这说明训练集和验证集虽然没有完全相同的文件，但同一原图的不同 Roboflow 变体大量分散在两个集合中。模型可能在训练阶段见过验证图片的近似版本，因此原验证指标不能作为可靠的竞赛结果。

## 3. 最终统一的类别定义

统一后的类别表如下：

| ID | 统一名称 | 中文含义 | 污染权重 | 板端策略 |
|---:|---|---|---:|---|
| 0 | `bird_drop` | 鸟粪 | 1.5 | 可清洁污染 |
| 1 | `clean` | 清洁表面 | 0.0 | 正常状态 |
| 2 | `dust` | 灰尘 | 1.0 | 可清洁污染 |
| 3 | `electrical_damage` | 电气损伤 | 0.0 | 故障告警，禁止自动清洁 |
| 4 | `physical_damage` | 物理损伤 | 0.0 | 故障告警，禁止自动清洁 |
| 5 | `snow_covered` | 积雪覆盖 | 0.0 | 环境告警，禁止湿洗 |

本次实现中，只有鸟粪和灰尘参与视觉脏污指数。电气损伤、物理损伤和积雪不计入自动清洁分数，而是进入更高优先级的安全限制。

## 4. 实际修改内容

### 4.1 新增板端统一类别策略头

新增文件：

```text
main/yolo_classes.h
```

该文件集中定义：

- 六个类别 ID。
- `YOLO_CLASS_COUNT` 和 `YOLO_NUM_CLASSES`。
- 鸟粪权重 `1.5`、灰尘权重 `1.0`。
- 类别告警优先级。
- 电气/物理损伤判断。
- 是否禁止湿洗的判断。

告警优先级为：

```text
电气/物理损伤 > 积雪 > 鸟粪/灰尘 > 清洁
```

这样可以避免多个目标同时出现时，仅按最高置信度目标控制设备。例如，同时检测到高置信度清洁板面和较低置信度物理损伤时，物理损伤仍应优先进入故障状态。

当前 LED 行为保持原有硬件意图：

- 清洁类别：不点亮污染/故障灯。
- 无检测结果：关闭两个状态灯。

### 4.4 增加视觉状态计算

修改文件：

```text
main/inference_task/inference_task.cpp
```

原代码只取检测结果数组中的第一个类别作为 `worst_class`。第一个结果通常只是置信度最高，并不一定是业务风险最高。

修改后会遍历所有检测结果并完成：

1. 按类别业务严重性选择最高优先级类别。
2. 仅对鸟粪和灰尘计算加权检测框面积。
3. 使用图像面积归一化，得到 `0-100%` 的视觉脏污指数。
4. 判断是否存在电气/物理故障。
5. 判断是否禁止湿洗。
6. 根据最高业务优先级控制 LED。
7. 通过串口日志输出视觉状态。

串口日志格式：

```text
Vision state: dirt=xx.x% fault=YES/NO wet_clean=BLOCKED/ALLOWED
```

这次只建立视觉侧的语义和安全基础，没有伪造尚不存在的完整执行器状态机，也没有接入电机或水泵控制。

### 4.6 重写数据集划分脚本

修改文件：

```text
yolov11/split_dataset.py
```

原脚本存在两个问题：

1. 使用随机图片级划分，没有约束同一原图的不同 Roboflow 变体。
2. 使用 `shutil.move()` 直接移动图片和标签，会破坏当前数据目录。

新脚本的处理流程：

1. 同时扫描现有 `train/` 和 `val/` 图片及标签。
2. 验证图片与标签是否一一对应。
3. 验证标签行是否为五列 YOLO 格式。
4. 验证类别 ID 和坐标范围。
5. 使用 `.rf.` 前的文件名作为原图来源 ID。
6. 将同一来源的所有变体组成不可拆分的组。
7. 使用固定随机种子 `42`。
8. 结合图片数量和多标签类别分布，贪心分配到 70%/15%/15%。
9. 只生成清单，不移动或复制原始图片和标签。
10. 自动检查三个集合的来源交集，发现泄漏时直接报错。

执行命令：

```powershell
cd D:\ESP32-IDF\esp32s3-cam\yolov11

```

### 6.3 构建产生的配置副作用

虽然构建目录指定在 `D:\tmp`，ESP-IDF 5.3.4 配置过程仍修改了仓库根目录：

```text

```

主要原因是仓库原配置由 ESP-IDF 5.4.2 生成，而本机导出的环境是 ESP-IDF 5.3.4。`dependencies.lock` 的 IDF 版本从 5.4.2 变为 5.3.4，`sdkconfig` 也被 5.3.4 重新规范化。

这两个文件不是类别统一功能的必要改动。在发布今晚的代码版本前，已经将它们恢复为仓库原有的 ESP-IDF 5.4.2 配置，不纳入代码提交。继续固件开发前，应统一团队使用的 ESP-IDF 版本，不能把两套版本生成的配置混合提交。

## 9. 后续建议

建议按以下顺序继续：

1. 决定项目统一使用 ESP-IDF 5.4.2 还是 5.3.4。
2. 在对应 IDF 环境中完成一次无超时的全量构建。
3. 使用 `yolov11/audit/samples.csv` 对每类 50 个框进行原图尺度复核。
4. 对确认错误的标签逐行修改，不进行未经验证的全局类别替换。
5. 使用 OV5640 采集清洁、灰尘、鸟粪和光照变化等真实场景。
6. 将真实设备数据保留为独立测试集，避免加入网络图片随机划分。
7. 上传数据和代码到云 GPU，先运行 1 个 epoch 验证环境和目录。
8. 再进行 60-100 epochs 正式训练。
9. 保存 `best.pt`、`last.pt`、`results.csv`、逐类指标和混淆矩阵。
