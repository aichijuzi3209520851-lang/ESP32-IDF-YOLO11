# DHT11 温湿度数据不显示 - 调试检查清单

## 【紧急】请按顺序提供以下信息

### 1. 串口日志（最重要）
运行 `idf.py -p COM16 monitor` 或在 VS Code ESP-IDF 插件里打开串口监视器，提供**最近 100 行日志**，特别关注：
- 包含 `DHT11` 的行（成功会显示温湿度值，失败会显示 timeout/checksum 错误）
- 包含 `CAM_APP` 的行（确认 dht11_task 是否正常启动）
- 包含 `ERROR` / `WARN` 的行

**示例：成功读取的日志长这样**
```
I (12345) CAM_APP: DHT11: 25.0C 60.0% RH
```

**示例：读取失败的日志长这样**
```
W (12345) DHT11: Sensor response timeout
W (12345) CAM_APP: DHT11 read failed: ESP_ERR_TIMEOUT
```

---

### 2. 浏览器开发者工具检查（如果串口日志显示读取成功）
按 F12 打开浏览器开发者工具，切到 **Network** 标签页，刷新页面后：
- 找到 `api/sensors` 这个请求
- 点击它，查看 **Response** 标签页里的 JSON 内容
- 截图或复制粘贴 JSON 结果

**期望看到：**
```json
{"temp":25.0,"humi":60.0,"ok":true}
```

**如果看到：**
```json
{"temp":null,"humi":null,"ok":false}
```
说明后端没拿到有效数据，需要回到步骤 1 检查串口日志。

---

### 3. DHT11 硬件接线确认
DHT11 模块通常有 3 个或 4 个引脚：

**3 脚模块（常见）：**
```
DHT11           ESP32-S3
─────────       ──────────
VCC (＋)   ──→  3.3V
GND (－)   ──→  GND
DATA/OUT   ──→  GPIO38
```
**板载上拉电阻的模块不需要额外电阻**。

**4 脚裸芯片（需要外接上拉）：**
```
DHT11           ESP32-S3
─────────       ──────────
VCC        ──→  3.3V
DATA       ──→  GPIO38 ──┐
NC         ──→  (空)      │
GND        ──→  GND       │
                          │
        4.7kΩ 上拉电阻 ──┘
       （DATA → 3.3V）
```

**请确认：**
- [ ] DHT11 的 VCC 接到了 ESP32-S3 的 3.3V（不是 5V）
- [ ] DHT11 的 GND 接到了 ESP32-S3 的 GND
- [ ] DHT11 的 DATA 接到了 ESP32-S3 的 GPIO38
- [ ] 如果是 4 脚裸芯片，DATA 和 VCC 之间接了 4.7kΩ 或 10kΩ 上拉电阻
- [ ] 如果是 3 脚模块（背面有小芯片），通常已经板载上拉电阻，不需要额外接

## 调试流程

```
┌─────────────────────────┐
│ 1. 检查串口日志         │
│    看 DHT11 是否正常读取│
└──────────┬──────────────┘
           │
           ├─ 日志显示 "DHT11: XX.XC XX.X%"  ──→ 驱动正常 ──→ 检查步骤 2（浏览器 API）
           │
           └─ 日志显示 "timeout" / "checksum" ──→ 硬件问题 ──→ 检查步骤 3（接线/上拉）
```

## 已排除的可能性（已验证）

✅ **代码编译** - 无报错，固件生成成功  
✅ **GPIO38 占用冲突** - ESP-IDF 官方头文件确认 GPIO38 空闲（Flash CS 是 GPIO29，PSRAM 占用 33-37，摄像头占用 4-18）  
✅ **HTTP API 实现** - `/api/sensors` 端点已正确注册，能返回 JSON  
✅ **前端轮询逻辑** - dashboard.html 每 3 秒调用一次 `fetch('/api/sensors')`  

❓ **待确认** - DHT11 硬件读取是否成功（需要串口日志）  
❓ **待确认** - 如果硬件读取成功，数据是否正确传递到 HTTP API（需要浏览器 Network 抓包）
